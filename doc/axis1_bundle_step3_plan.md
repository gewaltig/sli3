This document lists every operator in `src/builtins/` and classifies
it for the step-3 ABI conversion. The classification controls which
operators call `set_new_abi()` in their file's init function and
which keep the old ABI (default).

## Why classification matters

The dispatcher under the new ABI:

- **Main case** (`case sli3::functiontype:`): after `fn->execute`,
  check whether the e-stack top is still the fn slot (same
  `functiontype` tag AND same `func_val` pointer). If yes, pop;
  otherwise leave alone. **This handles old-ABI ops correctly**
  whether they self-popped, replaced their slot, or pushed iter
  frames — the post-check naturally distinguishes "slot still
  there" from "slot was mutated".
- **Iter cases** (`irepeat/iiterate/ifor/iforall`): if op is
  new-ABI, skip the sentinel push. If old-ABI, push the sentinel.
  **The sentinel is needed for ops that self-pop from inside an
  iter body** — without it, their self-pop would pop the
  iter_marker.

So `set_new_abi()` is the per-op opt-in that says: "I will not
self-pop and the dispatcher should handle slot cleanup
(main case) and skip the sentinel (iter case)."

## Three classes of operator

Per the count breakdown:

| File | trailing | mid-body | no-pop |
|------|---------:|---------:|-------:|
| sli_math.cpp | 25 | 48 | 24 |
| sli_typecheck.cpp | 9 | 5 | 2 |
| sli_control.cpp | 17 | 20 | 17 |
| sli_container_ops.cpp | 24 | 2 | 37 |
| sli_array_module.cpp | 3 | 6 | 9 |
| sli_io_ops.cpp | 2 | 0 | 7 |
| sli_startup.cpp | 6 | 1 | 6 |
| sli_state_ops.cpp | 0 | 2 | 0 |
| sli_builtins.cpp | 0 | 1 | 8 |
| sli_readline.cpp | 0 | 1 | 1 |

- **Trailing pop**: `i->EStack().pop()` is the last statement
  before the closing `}`. Safe to **remove**, set `new_abi`.
- **Mid-body pop**: `i->EStack().pop()` appears earlier in the
  body, followed by more code. **Inspection required** — could be
  any of:
  - "Early pop then math" pattern (math ops): operates on operand
    stack only after the pop. Safe to remove + new_abi (or
    just keep mid-body pop + new_abi; dispatcher's post-check
    handles both).
  - "Pop self then push proc" pattern (Ifelse, Case): replaces
    own slot. KEEP pop, leave old ABI. Don't set new_abi.
  - "Pop self then push iter frame" pattern (RepeatFunction,
    ForFunction, Loop, Forall_a, ...): same as above. Don't
    set new_abi.
- **No-pop**: op never touches `i->EStack().pop()`. Cases:
  - Stage 9 compaction dispatchers (Add, Sub, Mul, Div, Get,
    Put, ...): delegate to typed leaves via direct `.execute(i)`
    call. The leaf self-pops. Outer dispatcher's slot stays
    on top after call — but the leaf's self-pop has already
    consumed the OUTER's slot. So setting new_abi on the
    outer would cause double pop. **Keep old ABI.**
  - iparse / iparsestdin: stay-in-place re-entry pattern.
    KEEP old ABI.
  - Iter continuation operators (Iiterate, Iloop, Irepeat,
    Ifor, Iforall*): manage their own slot. KEEP old ABI.
  - Arraycreate / Dictconstruct: pop-to-mark patterns. KEEP
    old ABI.

### sli_container_ops.cpp (52 sites, 63 ops) — **MOSTLY CONVERTIBLE**

- 24 trailing → convert.
- 2 mid-body → inspect (likely cv* identity patterns).
- 37 no-pop → these are Stage 9 dispatchers (GetFunction,
  PutFunction, LengthFunction, ...) that call typed leaves via
  `leaf.execute(i)`. The leaf self-pops the OUTER's slot.
  **KEEP old ABI on dispatchers** — converting them would
  double-pop with the new dispatcher post-check.

**Wait, audit needed**: the Stage 9 dispatchers' pattern is:
```cpp
void GetFunction::execute(SLIInterpreter* i) const {
    // ... select arm ...
    if (...) { get_a_fn.execute(i); return; }
    // ...
}
```
`get_a_fn.execute(i)` runs synchronously. `get_a_fn` has its own
trailing self-pop (it's old-ABI today). It pops the OUTER's slot
(GetFunction's frame, which IS the top of e-stack right now —
the dispatcher placed it there).

Under new ABI for GetFunction: dispatcher pops after execute. But
get_a_fn already popped the slot. Dispatcher post-check: top is
NOT GetFunction → don't pop. ✓ correct.

So GetFunction-style dispatchers can be set_new_abi too. The
dispatcher's post-check handles it.

But — if `get_a_fn` is ALSO set_new_abi (and trailing pop
removed), then `get_a_fn` doesn't pop. After GetFunction's body
returns, top is still GetFunction → dispatcher pops. ✓ correct.

So whether to convert depends only on whether the LEAF is also
converted. Since Stage 9 dispatchers and leaves should convert
together, the answer is: **convert both** in one commit.

**Action:**

1. Remove the 24 trailing pops.
2. Inspect 2 mid-body pops.
3. Set new_abi on ALL container_ops (dispatchers + leaves) in
   one batch. The Stage 9 leaves (get_a_fn, put_a_fn, ...) are
   declared as anonymous-namespace statics in the .cpp; they
   register via `i->createcommand("get_a", &get_a_fn);`.

## Per-commit recipe

For each file `F`:

1. Identify the safe-to-convert set: ops with trailing pops OR
   "early pop then operand-stack work" (no e-stack push after).
2. Strip every `i->EStack().pop();` line that is the LAST
   statement before `}`:
   ```sh
   perl -i -0777 -pe \
     's/\n[ \t]+i->EStack\(\)\.pop\(\);[ \t]*(?:\/\/[^\n]*)?\n}/\n}/g' "$F"
   ```
   (The pattern allows a trailing `//` comment after the pop.)
3. For the converted ops, manually remove additional mid-body
   pops if they're "early pop then operand work" only. Skip if
   the pop is followed by push-to-e-stack.
4. In F's init function (e.g. `init_slimath`), generate
   `set_new_abi()` calls for every converted op:
   ```sh
   grep 'createcommand.*&' "$F" \
     | sed -E 's|.*&([a-z_]+function).*|    \1.set_new_abi();|' \
     | sort -u
   ```
   Filter the list to exclude any ops that should stay old ABI.
5. Build: `cmake --build build -j`
6. Run: `ctest --test-dir build --force-new-ctest-process`
7. Bench: `./build/sli3 < bench/sli/B*.sli` for affected
   workloads.
8. Commit.

## When step 3 is done

Step 3 is complete when every op in `src/builtins/` is either:
- Converted (trailing pop removed + `set_new_abi()` called), or
- Explicitly classified as "stays old ABI" (no `set_new_abi()`
  call, pops kept as-is).

Then step 4 flips the default to `new_abi_ = true` and removes
the old-ABI dispatcher path. Old-ABI ops would need explicit
`set_old_abi()` calls. Or — much simpler — step 4 also converts
those remaining ops by changing the dispatcher to handle their
patterns without the sentinel.

## Things to watch for during conversion

- **Test failures with "stop FATAL: No stopped context"**: an op
  is calling raiseerror via a path that doesn't unwind cleanly.
  Usually means an op was converted that shouldn't have been
  (iter setup, slot replacement).
- **test_dispatch_parity failures**: the parity test compares
  plain (`execute_`) vs dispatch (`execute_dispatch_`). Plain
  doesn't run the dispatcher's new-ABI code path. Some divergence
  is expected for converted ops (plain still uses old ABI via
  SLIType::execute). Document if it diverges.
- **test_features failures with stack depth**: usually means an
  op was converted but its mid-body push-to-estack wasn't
  accounted for.
- **Infinite loops / timeouts (`test_math_overflow timeout`)**:
  an op was set_new_abi without removing its self-pop, but the
  self-pop is mid-body in a way that... actually, this can't
  cause an infinite loop. More likely: an op WASN'T set_new_abi
  but its trailing pop was removed → it doesn't pop self, and the
  dispatcher doesn't either → fn slot stays on top → dispatcher
  re-dispatches the same fn forever. **The fix: every removed
  pop MUST be paired with a set_new_abi.**

## Step 4 (after step 3 complete)

Flip the default and clean up:

1. `SLIFunction::new_abi_` default true.
2. Drop the `if (fn->uses_new_abi())` branches in the dispatcher;
   the new-ABI path is the only path.
3. Drop the sentinel push from iter cases (was only there for
   old-ABI ops).
4. Old-ABI special ops (the ones I excluded from set_new_abi
   above) need to be made compatible with the new dispatcher.
   Most options:
   - **Inspect and rewrite each** to fit the new contract.
     Tedious but clean.
   - **Keep the dual-ABI dispatcher as a permanent feature** if
     the special ops are stable. The flag becomes "this op
     needs the sentinel" rather than transitional.

For now, plan for the latter: keep the flag and the dual-ABI
dispatcher. Step 4 just removes the unused `set_new_abi()`
boilerplate (most ops keep the flag set; a small set explicitly
call `set_old_abi()`).
