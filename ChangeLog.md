- **2026-05-22 arraytype — re-introduce C++ Map / MapIndexed / MapThread** (commit `78e39b1`, run_id 45).   - **Algorithm (per user spec, May 2026).** Entry op clones the source array into `target` on the estack and **leaves the original on the operand stack** so a body-raised error unwinds with the original recoverable. Each iter pushes the next element (Map) / element+1-based-index (MapIndexed) / one-element-per-row tuple (MapThread) onto the ostack, executes the body via the dispatcher's body-walk, checks `ostack.load() == saved_load + 1` (StackUnderflow if violated — sanity check on user procs that consume too much or push too little), then writes back `target[idx-1] = ostack.top()` and pops. Final iter overwrites the original on the ostack with `target` and pops the estack frame. This differs from NEST 2.20.2's `MapFunction` (which *moves* the array off the ostack at entry); preserving the original lets `stopped`/`raiseerror` recover the user's input on a mid-iter failure.

- **2026-05-23 `/return` — Python-style early procedure exit** (commits `e1c1bbb`, `7bd39c3`, `c763789`). New control-flow operator that abandons the rest of the enclosing procedure body and any loops between the call site and that procedure. Closes a long-standing usability gap where the only way to exit a search loop in SLI was an `/exit` followed by a flag-check on the post-loop value, or a `{ stopped { } if }` wrapper.
  - **Algorithm.** `ReturnFunction::execute` walks the e-stack from the top, stepping past loop frames by their known sizes (iforall/iforall_s 6, ifor/iforallindexed_a/iforallindexed_s/imap/imapindexed 7, iloop 4, irepeat 5, imapthread 8). Two terminations: hit `iiteratetype` → pop `n+3` (marker + pos + proc). Hit a non-iter slot after stepping past ≥1 loop frame → pop `n` (the loop frames the dispatcher's single-level iiterate TCO has already collapsed the enclosing procedure away for, e.g. a body ending in `… forall_a`). Hit a non-iter slot with `n == 0` → raise `InvalidReturn`. This makes the common `{ … forall { … return } }` pattern Just Work even when the enclosing proc has been TCO'd.
  - **Interaction with TCO.** The dispatcher's single-level iiterate TCO collapses a procedure's iiterate frame *before* its last token runs. Side effect (and a feature, not a bug): a tail `/return` inside a one-token if-branch (`{ return }`) unwinds the *outer* proc, mirroring Python's `if cond: return`. A mid-body `/return` (something follows in the branch) keeps the branch frame intact and unwinds only the branch. Both shapes are documented in `lib/sli/sli-init.sli` and in the tutorial.
  - **Registration.** `createcommand("return", &returnfunction)` alongside `/exit` in `init_slicontrol` (`sli_control.cpp:2098`); class declared in `sli_control.h` next to `ExitFunction`.
  - **Tests.** Nine functional cases in `tests/test_sli_eval.cpp` covering mid-body return, return from inside `/forall_a` / `/for` / `/repeat` bodies, nested loops collapsing together, `/exec` mid-body return, and both the TCO and non-TCO `/if`-branch interactions. One error case in `tests/test_errors_dispatch.cpp` confirms `InvalidReturn` is raised when `/return` is called outside any procedure. 23/23 ctests green on Release.
  - **Documentation.** `lib/sli/sli-init.sli` `@BeginDocumentation` block in the `/bind` / `/length` style with synopsis, mechanics, the TCO interaction note, examples (`findFirst` pattern), and `SeeAlso: exit, stop, stopped, exec`. New section in `doc/tutorial/05_loops.md` with a contrast against `/exit` — both snippets verified against `build/sli3`. Row in `doc/tutorial/core_commands.md`. Side-by-side recursive Fibonacci example (`/return` form vs. nested `/ifelse` form) in `doc/tutorial/examples/ch05_loops/fib_return.sli` showing the readability win.
