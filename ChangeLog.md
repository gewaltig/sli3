- **2026-05-22 arraytype — re-introduce C++ Map / MapIndexed / MapThread** (commit `78e39b1`, run_id 45).   - **Algorithm (per user spec, May 2026).** Entry op clones the source array into `target` on the estack and **leaves the original on the operand stack** so a body-raised error unwinds with the original recoverable. Each iter pushes the next element (Map) / element+1-based-index (MapIndexed) / one-element-per-row tuple (MapThread) onto the ostack, executes the body via the dispatcher's body-walk, checks `ostack.load() == saved_load + 1` (StackUnderflow if violated — sanity check on user procs that consume too much or push too little), then writes back `target[idx-1] = ostack.top()` and pops. Final iter overwrites the original on the ostack with `target` and pops the estack frame. This differs from NEST 2.20.2's `MapFunction` (which *moves* the array off the ostack at entry); preserving the original lets `stopped`/`raiseerror` recover the user's input on a mid-iter failure.
